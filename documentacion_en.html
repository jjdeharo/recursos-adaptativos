1. Purpose of the protocol

This document serves as a guide for creating adaptive educational applications, activities or quizzes based on Bayesian inference and Shannon entropy.

The objective is not to create a linear test or a rigid sequence of questions, but rather a system capable of adapting the student's experience based on their answers. Each response is interpreted as evidence that progressively modifies a probability distribution on different educational hypotheses.

These hypotheses may refer to:

  • level of conceptual mastery;
  • probable errors;
  • previous ideas;
  • specific gaps;
  • reinforcement needs;
  • preparation to advance to a higher level.

The system must produce a pedagogical interpretation, not just a score.

2. General principle

The system must use the student's responses as evidence to update hypotheses about their learning situation.

Each response should modify the estimate gradually. A single answer should not determine the outcome by itself.

A correct answer should increase the likelihood of certain hypotheses and an incorrect answer should increase the likelihood of others, always according to the likelihoods associated with each question.

The system must avoid definitive conclusions when uncertainty remains high.

3. Representation of the student's status

The student's state should be represented as a probability distribution over several hypotheses.

For example, a simple system might use three hypotheses:

  • basic level;
  • intermediate level;
  • advanced level.

But the system should not necessarily assume three levels. It must be able to adapt to more or fewer levels, depending on the educational context.

Non-strictly hierarchical hypotheses can also be used, for example:

  • understand the concept;
  • confuses two closely related ideas;
  • applies a rule mechanically;
  • has a prior gap;
  • needs procedural reinforcement;
  • can move to an extension task.

If there is no prior information about the student, the system can begin with balanced probabilities. If reliable prior information exists, a justified initial distribution can be used instead.

3.1. Multidimensional state

The student's state does not have to be represented by a single distribution. When it is useful to know not only how much has been mastered, but also which specific skills or steps are failing, the system can maintain several probability distributions in parallel: one per category or level and another for each diagnostic dimension (a skill, a process step, or a type of error).

All of them are updated from the same student response: the overall result modifies the belief about level, and each part of the response modifies the belief about the corresponding dimension. In this way, the level estimate indicates what should be practised, while the diagnosis by dimension indicates what should be explained or reinforced. Each dimension may have its own probability of success by chance, so percentages from different dimensions are not directly comparable; the shared reference is the underlying latent level.

4. Bayesian update

The system must update the probability distribution using Bayesian inference.

For each hypothesis, the probability of observing the student's response if that hypothesis were true must be estimated.

That probability is the likelihood.

If the student answers a question correctly, the system must use the probability of success under each hypothesis. If the answer is incorrect, it must use the probability of failure under each hypothesis.

The result of each update must be a new probability distribution on the considered hypotheses.

The process must be repeated after each response.

5. Basic model formulas

Although the document should remain understandable for teachers, it is advisable to include the essential formulas so that the AI can implement the system coherently.

5.1. Initial distribution of hypotheses

If there are \(n\) hypotheses and no reliable prior information exists, a uniform distribution can be used:

\[P(H_i)=\frac{1}{n}\]

Here \(H_i\) represents one possible hypothesis about the student's state.

5.2. Bayesian update

After each answer, the probability of each hypothesis is updated by:

\[P(H_i\mid R)=\frac{P(R\mid H_i)P(H_i)}{P(R)}\]

The denominator is calculated by adding over all the hypotheses:

\[P(R)=\sum_i P(R\mid H_i)P(H_i)\]

Here \(R\) is the observed response, which may be correct, incorrect, or another automatically scorable result anticipated by the system.

5.3. Probability of success and failure

For each question \(q\) and each hypothesis \(H_i\), the system must estimate:

\[P(\text{correct}\mid H_i,q)\]

If the student answers incorrectly, the system uses:

\[P(\text{incorrect}\mid H_i,q)=1-P(\text{correct}\mid H_i,q)\]

These probabilities are the likelihoods that fuel Bayesian updating.

5.4. Minimum probability of success by chance

In multiple choice questions, the minimum probability of being correct by chance depends on the number of options in each question:

\[c_q=\frac{1}{m_q}\]

Here \(c_q\) is the probability of answering question \(q\) correctly by chance, and \(m_q\) is the number of options in that question.

For example, a four-option question has:

\[c_q=\frac{1}{4}=0.25\]

This probability should belong to each question, not to the entire test.

5.5. Logistic generation of likelihoods

The system should be able to generate these likelihoods automatically. A recommended option is to use a logistic function adjusted for chance:

\[P(\text{correct}\mid H_i,q)=c_q+(1-c_q)\cdot\frac{1}{1+e^{-a(\theta_i-b_q)}}\]

Where:

  • \(\theta_i\) numerically represents the hypothesis or level \(H_i\);
  • \(b_q\) represents the difficulty of question \(q\);
  • \(a\) controls sensitivity or discrimination;
  • \(c_q\) is the minimum probability of success by chance.

This formula does not replace Bayes. It only generates the likelihoods that Bayes needs.

The \(a\) parameter controls the slope of the logistic curve. A high value makes the function discriminate more between close hypotheses; a lower value produces smoother transitions. The usual values in psychometrics range between 0.5 and 2.5. For general purpose educational systems, a value of 1.0 to 1.5 is a reasonable starting point; 1.5 is a good default choice.

5.6. Shannon entropy

The uncertainty of the system is measured by:

\[H=-\sum_i p_i\log_2(p_i)\]

Where \(p_i\) is the current probability of each hypothesis.

The maximum entropy, when all hypotheses are equally probable, is:

\[H_{\max}=\log_2(n)\]

This value helps adjust the stopping threshold to the number of hypotheses considered.

5.7. Expected information gain

To select the next question, the system can estimate the expected reduction in uncertainty:

\[IG(q)=H(\text{before})-\left[P(A)\,H(\text{after correct})+P(F)\,H(\text{after incorrect})\right]\]

Where \(P(A)\) is the total expected probability of success and \(P(F)\) is the total expected probability of failure, calculated using the law of total probability:

\[P(A)=\sum_i P(H_i)\cdot P(\text{correct}\mid H_i,q)\]
\[P(F)=1-P(A)\]

The posterior distributions for a correct and an incorrect response are calculated by applying Bayes before the actual answer is known:

\[P(H_i\mid\text{correct})=\frac{P(\text{correct}\mid H_i,q)\,P(H_i)}{P(A)}\qquad P(H_i\mid\text{incorrect})=\frac{P(\text{incorrect}\mid H_i,q)\,P(H_i)}{P(F)}\]

When possible, the most useful question will be the one that produces the greatest expected reduction in entropy.

5.8. Likelihood with partial credit

The formulas above assume that the response is binary: correct or incorrect. But many tasks are solved through steps or components and allow partial credit. In those cases, instead of classifying the response as simply correct or incorrect, it is summarised as a score \(s\) between 0 and 1, and the likelihood is obtained by interpolating between success and failure:

\[L(H_i)=s\,P(\text{correct}\mid H_i,q)+(1-s)\,P(\text{incorrect}\mid H_i,q)\]

If \(s=1\), it is equivalent to a completely correct answer; if \(s=0\), to a completely incorrect answer; if \(s=0.5\), the likelihood is flat and does not modify the distribution. This soft likelihood enters Bayesian updating just like any other: it only changes the way it is constructed. The \(s\) score must be calculated explicitly and automatically, usually as a weighted sum of subcriteria whose weights add up to 1. The formal development and a numerical example are in the mathematical explanation.

6. Likelihoods generated by the system

Likelihoods are the central element of the system.

For each question, the system must estimate:

  • what probability a student in each hypothesis would have of answering it correctly;
  • what probability a student in each hypothesis would have of answering it incorrectly.

The teacher should not have to manually fill out a probability table. That task must be done by the system.

The teacher should only provide understandable pedagogical information, such as:

  • number of levels or hypotheses;
  • relative difficulty of each question;
  • question type;
  • number of options, if applicable;
  • concept evaluated;
  • possible associated conceptual error.

From this information, the program must automatically generate the likelihoods. This can be done with a generative model, preferably a chance-adjusted logistic function, and the values can later be recalibrated with real data if enough responses are available.

The recommended workflow is therefore:

  1. The teacher designs or reviews the questions and assigns them difficulty and format.
  2. The system automatically calculates the probabilities of success and failure for each hypothesis.
  3. Bayes uses these probabilities as likelihoods.
  4. If real data accumulates, the system can adjust its parameters to improve calibration.

Teacher judgment should guide the definition of levels, difficulties, objectives, and concepts, but it should not require teachers to enter numerical probabilities.

7. Dynamic likelihoods per question

The application should not depend on a fixed global likelihood table.

Each question should be able to have its own parameters, especially:

  • estimated difficulty;
  • question type;
  • number of options, if applicable;
  • minimum probability of success by chance;
  • concept or skill assessed;
  • possible associated conceptual error.

This allows the system to work with questions of different types within the same test.

For example, a test may include two-option, four-option, five-option, matching, and numerical-response questions. Each one must generate its own likelihoods.

8. Difficulty of the questions

Each question must have an estimated difficulty.

Difficulty can be expressed qualitatively, for example as easy, medium, or difficult. It can also be expressed on a more flexible scale.

The system should not assume that there will always be three difficulties. It must adapt to the number of categories defined by the teacher.

When the teacher defines difficulty qualitatively, the system must convert it into numerical values so that it can work with the logistic function. A useful convention is to center the values at zero and space them at equal intervals:

Difficulty CategoriesNumerical values \(b_q\)
2−0.5 · · 0.5
3−1 · · 0 · · 1
4−1.5 · · −0.5 · · 0.5 · · 1.5
5−2 · · −1 · · 0 · · 1 · · 2

The \(\theta_i\) numerical values of the levels or hierarchical hypotheses follow the same convention of centering at zero with equal intervals, but they must cover a range strictly greater than that of \(b_q\). This condition is important: if \(\theta_{\max} = b_{\max}\), the extreme level and the extreme difficulty coincide at the inflection point of the logistic curve. At that point the question is very discriminating, but the probability of correctness at the extreme level can still be moderate, so getting it right weakly confirms that level. A factor of 2 works well in practice: \(|\theta_{\max}| = 2 \cdot |b_{\max}|\). For example, with 3 \(b \in \{-1, 0, +1\}\) difficulties and 3 levels, using \(\theta \in \{-2, 0, +2\}\) makes difficult questions more informative for advanced learners. The most robust thing is to calculate \(\theta_{\max}\) automatically: \(\theta_{\max} = 2 \cdot \max|b_q|\).

9. Variable number of levels and hypotheses

The system must allow different numbers of hypotheses:

  • two states: masters / does not master;
  • three levels: basic / intermediate / advanced;
  • four levels: initial / basic / competent / advanced;
  • alternative conceptual hypotheses within the same dimension: error A / error B / correct understanding;
  • need-based hypotheses, provided that they are alternatives within the decision being modeled: prior reinforcement / guided practice / consolidation / extension.

The entropy threshold and stopping criteria must be adapted to the number of hypotheses considered. The same threshold should not be used for all designs without justification.

If you want to stop when the most probable hypothesis exceeds a \(p_{\min}\) confidence level (for example, 0.80), an indicative entropy threshold associated with that confidence level is:

\[H_{\text{stop}}=-p_{\min}\log_2 p_{\min}-(1-p_{\min})\log_2\!\left(\frac{1-p_{\min}}{n-1}\right)\]

This formula assumes that the remaining probability is distributed evenly among the other \(n-1\) hypotheses. Some guide values with \(p_{\min}=0.80\):

Hypothesis \(n\)\(H_{\text{stop}}\) (bits)
20.72
30.92
41.04
51.12

The above formula assumes that the remaining probability is distributed evenly among the other \(n-1\) hypotheses, which does not always happen in real distributions. For example, with three hypotheses, the distributions (0.80, 0.10, 0.10) and (0.80, 0.19, 0.01) have the same maximum hypothesis but different entropy. Therefore, the \(H_{\text{stop}}\) threshold should be understood as a practical approach, not as an exact equivalent. In the implementation, it is advisable to check both criteria in a complementary way: entropy below the threshold and most probable hypothesis above \(p_{\min}\).

The hypotheses of the same distribution must be mutually exclusive and cover the relevant alternatives for that decision. If multiple errors, gaps, or needs can coexist, it is a good idea to use separate distributions by concept, dimension, or category, rather than forcing them into a single list of hypotheses.

When the hypotheses are not hierarchical—alternative categories with no order relationship between them, such as different conceptual errors, resolution strategies the student uses, possible causes of the same error, or thematic areas with no hierarchy—the IRT logistic function may not be the most appropriate model, because it assumes a single scale of “more or less mastery”. In these cases it is better to define the likelihoods in another way: for example, by directly assigning a high probability of success to questions that diagnose the correct hypothesis and a lower probability to those that confuse it with other hypotheses. Bayesian updating remains identical; only the way the likelihoods are generated changes.

To assign those probabilities without empirical data, an explicit criterion helps. For each question and each hypothesis, estimate: "if the student had this error, how often would they answer this question correctly?". The value will be low when the question attacks precisely the concept the error distorts, and high when the error does not interfere. Bound each value between a guessing floor (no less than \(1/m\), with \(m\) options) and a ceiling below 1 for the mastery hypothesis (around 0.90–0.95, leaving room for slips). If the question is multiple-choice and one of the distractors is precisely the answer the error produces, the student is drawn toward that option and their probability of success falls below chance. There is no need to fine-tune the exact decimal: it is enough that, in discriminating questions, the correct hypothesis has clearly more probability than the one the error makes fail. The mathematical formulation (§10.5) details these buckets and their validation.

10. Types of acceptable questions

If the system must work automatically, it should use automatically scorable question types.

Suitable formats include, among others:

  • multiple choice with one correct answer;
  • true/false;
  • multiple selection with clear correction criteria;
  • matching;
  • ordering;
  • numerical response with tolerance;
  • exact short answer.

Long open-ended questions should not be included if the program cannot reliably correct them automatically.

Open questions can be useful in a general educational activity, but they should not be part of the automatic adaptive engine if there is no reliable mechanism for correction or teaching intervention.

11. Probability of success by chance

In multiple choice questions, the minimum probability of being correct by chance must depend on the number of options in each question. This probability belongs to each item, not to the entire test.

Examples:

  • 2-option question: minimum probability of success by chance of 50%;
  • 3-option question: minimum probability of success by chance of 33.3%;
  • 4-option question: minimum probability of success by chance of 25%;
  • 5-option question: minimum probability of success by chance of 20%.

If within the same test there are questions with a different number of options, each question must use its own minimum probability of success by chance.

In multiple selection with several correct answers, the probability of success by chance may be different. The system must handle this case specifically, especially if exact matching with the correct combination is required.

In numerical responses with tolerance or exact short responses, the probability of correctness by chance may be considered null or very low, depending on the design.

When an item is scored through several components with different numbers of options—for example, a task with several steps—its probability of correctness by chance is not that of a single question. In that case it is calculated as the weighted average of the chance levels of each component, using the same weights with which the response is scored. This aggregated value is what feeds the logistic function of the complete item.

12. Use of a logistic function

Where appropriate, likelihoods can be generated using a chance-adjusted logistic function.

The general idea is that the probability of correctness should increase when the student's hypothetical level exceeds the difficulty of the question, and should decrease when the difficulty exceeds the student's hypothetical level.

If this model is used, an important condition must be respected: the probability of correctness must not be less than the probability of correctness due to chance inherent to the question.

Therefore, a four-option question should not assign a probability of success less than 25%, because a student who answers at random has that probability of being correct.

The logistic function does not replace Bayes. It only serves to generate the likelihoods that Bayes needs.

The system must allow this generation to be adjustable. For example, there may be a sensitivity or discrimination parameter that determines how much a question separates between some hypotheses and others.

13. Adaptive selection of the next question or activity

From the current estimate, the program should dynamically select the next question, explanation or activity.

The selection should aim at pedagogical usefulness and may serve to:

  • confirm a probable hypothesis;
  • distinguish between close hypotheses;
  • detect a conceptual error;
  • check a specific gap;
  • reinforce a difficulty;
  • cover content not yet assessed;
  • allow the student to demonstrate a higher level;
  • decide whether reinforcement or extension is more appropriate.

The system should not limit itself to raising the difficulty after a success and lowering it after a failure.

The system should take into account response history, concepts already assessed, detected errors, content variety, relative question difficulty, question type, number of options in each question, probability of success by chance, and the confidence level of the current estimate.

When several questions have the same expected information gain, which often happens when they share the same difficulty parameters and number of options, the choice among them should not be deterministic. A weighted random selection is recommended: calculate the gain for all candidates, gather those within a small margin of the maximum, and choose among them with probability inversely proportional to how often their category or concept has already appeared. This combines maximum informative usefulness with thematic diversity, without imposing rigid restrictions. Deterministic tie-breaking leads to systematically repetitive tests across different sessions.

13.1. Initial diagnosis and subsequent reinforcement

In adaptive practice, reinforcement, or skill-training resources, selection should not always pursue the same objective. At first it is useful to diagnose; afterwards it is better to intervene where mastery is weakest.

A two-phase strategy is recommended:

  1. Initial diagnostic phase. As long as there are categories, concepts, or problem types with less than a minimum sample of attempts, the system must prioritize them. The goal is to avoid conclusions based on too small a sample. Within these categories, the question with the greatest expected gain of information can be chosen. A reasonable default is to require at least 2 attempts per category.
  2. Reinforcement phase. When all relevant categories have a minimum sample, the system must prioritize the category with the lowest estimated mastery or greatest pedagogical need. Once the category is chosen, the specific question does not necessarily have to be the most difficult. It is advisable to combine the expected gain of information with a measure of difficulty adequacy, so that the system prefers informative exercises but close to the student's estimated level.

This separation avoids excessive use of Shannon entropy. Entropy answers the question “where do I have the most uncertainty?”, but it does not always answer “what does the student need to practice the most?” In pure diagnostic evaluation, maximizing uncertainty reduction may be the main criterion. In adaptive practice and reinforcement, it should be combined with a pedagogical utility function that gives priority to less mastered content.

Operationally, the reinforcement phase can use a utility score such as:

utility = α · Normalized_GI + (1 − α) · difficulty_adjustment

where difficulty_adjustment decreases when the question difficulty moves too far away from the student's estimated level. A reasonable initial value is α = 0.6 or 0.7, so as to preserve diagnostic value without turning reinforcement into a succession of overly difficult exercises.

13.2. Continuous reinforcement and caution with little evidence

The stopping criteria (section 17) assume that the goal is to diagnose and stop. In open practice or reinforcement resources, however, there may be no fixed endpoint: the session continues while the student practises, and the estimated state should be understood as an evolving estimate, not as a closed diagnosis. In that mode, entropy and confidence indicate the degree of certainty rather than ending the session, and the two-phase selection strategy remains active throughout the practice.

With only a few attempts, a single answer can shift the estimate substantially. To avoid communicating unfounded mastery, it is advisable to require a minimum sample per category or dimension before showing high levels of mastery, and to present the estimate as provisional while the evidence remains scarce. This is a presentation decision: it does not change the Bayesian calculation, only how it is reflected in the interface.

14. Expected reduction in uncertainty

When possible, the system should select the question that most reduces the expected uncertainty.

To do so, the system must estimate, before presenting the question:

  • what would happen if the student gets it right;
  • what would happen if the student fails;
  • how the probability distribution would change in each case;
  • what entropy each posterior distribution would have;
  • what would be the expected uncertainty after asking that question.

The best question does not have to be the one that matches the most probable level. It can be a question that helps distinguish between two still plausible hypotheses.

For example, if the system hesitates between intermediate and advanced levels, a difficult question may be more useful than a medium one. If it hesitates between basic and intermediate, a medium or easy question may be more informative, depending on the likelihoods.

15. Recovery mechanisms

When the selection of questions is made based on the maximum expected information gain, recovery remains largely integrated in the Bayesian mechanism itself: if the student initially fails but then correctly answers more difficult questions, the posterior is automatically moved and the system selects more demanding questions. Explicit retrieval logic is usually not necessary, but full retrieval is not guaranteed if the questions are poorly calibrated, if few are available at any level, or if the student responds at random.

Explicit recovery mechanisms are only necessary when question selection is based on simple difficulty rules (up after a correct answer, down after an incorrect one), because in that case the system may get stuck at the wrong level. If selection by information gain is used, the risk of blocking is greatly reduced, because the system reevaluates the posterior after each response. Still, practical blocking can occur if the question bank is limited, if the likelihoods are poorly calibrated, or if the test is stopped too early.

What must be guaranteed in any design:

  • The system should not lock the student irreversibly into an initial category.
  • If the student fails at first but then shows consistent signs of mastery, the Bayesian posterior will reflect that change.
  • If the student starts with correct answers but then shows relevant errors, the system will review the probability distribution.
  • The adaptation must be dynamic in each response, not just at the beginning or the end.

16. Shannon entropy as a measure of uncertainty

A reasonable degree of confidence should be estimated through Shannon entropy applied to the probability distribution over the hypotheses.

When probabilities are widely spread, entropy is high and the system remains uncertain. When one hypothesis concentrates most of the probability, entropy is low and the system is more confident.

Entropy must serve to:

  • decide if it is appropriate to continue asking;
  • compare uncertainty before and after a question;
  • select informative questions;
  • determine if the final diagnosis is firm or provisional.

The stopping threshold must be adjusted to the number of hypotheses considered and the desired degree of precision.

17. Completion criteria

The process should end when there is reasonable confidence about the student's learning state or when a practical limit is reached.

Possible stopping criteria are:

  • The entropy falls below the threshold \(H_{\text{stop}}\) and the most probable hypothesis exceeds \(p_{\min}\). Both criteria are complementary and it is advisable to check both: \(H_{\text{stop}}\) is calculated assuming uniform distribution of the remainder, which is an approximation (see §9).
  • A maximum number of questions is reached. It is useful to distinguish between a theoretical maximum and a practical maximum. The former can be estimated by minimax search on the tree of possible responses; the latter should be set for usability. In a demo or short educational tool, for example, a hard limit of 20 questions may be imposed, showing a provisional result if the system has not converged earlier.
  • The best question available is no longer useful. If, after a sufficient number of questions, the expected information gain of the best candidate falls below a minimum threshold, it is advisable to stop the session and return a provisional diagnosis instead of artificially lengthening it.
  • The minimum planned concepts have already been covered.
  • There are no useful questions available.
  • The cost of continuing to ask exceeds the expected pedagogical gain.

If the final entropy is still high, the system should clearly indicate this. In that case, the result should be presented as a provisional estimate, not as a firm conclusion.

18. Final result

The final result must be presented as a pedagogical interpretation.

Where appropriate, it should include:

  • what the student seems to master;
  • what difficulties the student appears to show;
  • what misconceptions are likely;
  • what gaps should be reviewed;
  • what type of reinforcement is recommended;
  • what advancement activity might be proposed;
  • how confident the estimate is;
  • whether the diagnosis is firm or provisional.

It should not be limited to displaying a score or a label.

The purpose of the system is to help make educational decisions.

19. Adaptation to the teaching context

The concept of an adaptive educational resource should be understood broadly. An adaptive test is only one particular case. The same logic can be applied to explanations, practice activities, simulators, learning pathways, hint systems, reinforcement activities, extension activities, or resource recommendation tools. What matters is that the resource makes pedagogical decisions on the basis of the evidence gathered during the interaction with the student.

The concrete implementation must adapt to the educational context indicated by the teacher.

Before designing the system, it is advisable to collect information about:

  • topic or teaching unit;
  • age or grade of the students;
  • learning objectives;
  • number of levels or hypotheses;
  • types of questions or interactions allowed;
  • approximate number of questions or steps;
  • maximum duration;
  • type of adaptive resource desired (evaluation, practice, itinerary, tutorial, simulator, game, recommender, review...);
  • main purpose (detect level, identify errors, reinforce, practice, consolidate, expand, guide, recommend...);
  • what the system should adapt (difficulty, type of activity, explanation, help, clues, rhythm, itinerary...);
  • type of expected output (diagnosis, progress report, learning path, proposal for the next activity...);
  • desired format;
  • necessary degree of precision.

The system must adapt its decisions to that context.

21. Design rules for AI

When generating an application or activity based on this protocol, the AI must follow these rules:

  1. Do not create a rigid path or a linear test.
  2. Don't limit yourself to increasing difficulty after success and lowering it after failure.
  3. Represent the student's state using probabilities on hypotheses.
  4. Update those probabilities gradually.
  5. Dynamically generate likelihoods, without requiring the teacher to enter probability tables.
  6. Take into account the difficulty of each question.
  7. Take into account the number of options for each question.
  8. Do not use long open questions if they cannot be corrected automatically.
  9. Select questions by pedagogical usefulness and expected information.
  10. Use Shannon entropy to measure uncertainty.
  11. Allow recovery from wrong initial estimates.
  12. Present the result as a pedagogical interpretation.
  13. Indicate when the estimate is provisional.
  14. Adapt to the context provided by the teacher.
  15. Avoid unnecessary technical details if the user only needs methodology.

23. Scope and limits

This protocol does not replace teacher judgment.

The system can help guide decisions, but its results should be interpreted with caution, especially when:

  • there are few questions;
  • the questions are not well calibrated;
  • the initial likelihoods come from a generating model not calibrated with real data;
  • the student responds at random;
  • the application context is very limited;
  • the final entropy is still high.

The main value of the approach is in making uncertainty explicit and in adapting the activity to the available evidence.

When the result is to be used for decisions, it is advisable to accompany it with reliability checks that require no empirical data: a pattern-fit index for each student (it detects responses inconsistent with difficulty, which would make the diagnosis unreliable even when confidence is high) and a validation of the design by simulation (it estimates, before applying the test, to what extent the bank separates the levels). Both are described in the mathematical foundations (§11.7–§11.8); they are an optional layer, not a mandatory step, and they measure reliability under the model, not empirical validity.

In extended learning resources such as tutorials, adaptive practice, reinforcement, or extension activities, the student's state may change during the session itself. In such cases, the posterior should not be interpreted only as the diagnosis of a fixed state, but as a dynamic estimate that can combine recent evidence, support used, observed progress, and changes in performance.
